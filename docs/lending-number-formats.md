# Lending number formats

Reference for the human-readable identifiers used across the SACCO
platform. Every external-facing surface (admin UI, exports, member
statements, regulator reports) should use these — never the underlying
UUID.

All formats are **per-tenant sequential** with a 4-digit year prefix.
The sequence resets on the first record of each calendar year so
year-end audit reports get clean ranges. UUIDs remain the
internal-only join key.

## Active formats

| Domain | Format | Example | Sequence source |
|---|---|---|---|
| Member (the person) | `M-YYYY-NNNNN` | `M-2026-00003` | `member_number_seq` per tenant |
| Member onboarding application | `APP-YYYY-NNNNNN` | `APP-2026-000003` | `membership_application_seq` per tenant |
| Loan application | `LA-YYYY-NNNNN` | `LA-2026-00008` | `loan_application_seq` per tenant |
| Disbursed loan | `L-YYYY-NNNNN` | `L-2026-00003` | `loan_number_seq` per tenant |
| Share account | `SHA-YYYY-NNNNN` | `SHA-2026-00003` | `share_account_seq` per tenant |
| Share transaction | `SHT-YYYY-NNNNN` | `SHT-2026-00007` | `share_txn_seq` per tenant |
| Deposit account | `DPA-YYYY-NNNNN` | `DPA-2026-00005` | `deposit_account_seq` per tenant |
| Deposit transaction | `DPT-YYYY-NNNNN` | `DPT-2026-00019` | `deposit_txn_seq` per tenant |
| Loan transaction | `LT-YYYY-NNNNN` | `LT-2026-00044` | `loan_txn_seq` per tenant |
| GL journal entry | `JE-NNNNNN` | `JE-000050` | `journal_entry_seq` per tenant (no year prefix — entries are dated by header) |
| Member status change | `MSC-YYYY-NNNNNN` | `MSC-2026-000012` | `member_status_change_seq` per tenant |

## Why two "application" formats

The IA used to call both the membership-join queue and the loan-app
queue "Applications", and the sidebar label was just "Applications" —
ambiguous. As of this commit:

- The membership-join queue lives at `/applications` and is labelled
  **"Member onboarding"** in the sidebar + page header. Its records use
  `APP-YYYY-NNNNNN`.
- Loan applications live under `/loans` → the **"Loan applications"**
  sub-tab. Their records use `LA-YYYY-NNNNN`.

`APP-` and `LA-` are intentionally distinct prefixes so a number on its
own (in an email, an audit log entry, or a regulator report) is
self-identifying. Don't reuse `APP-` for a non-membership-onboarding
context.

## Disallowed / legacy

| Pattern | Where it appears | Status |
|---|---|---|
| `PROV-W-…` / `PROV-S-…` / `PROV-L-…` (with unix-ts suffix) | Test seed scaffold for the provisioning module — 3 fixed loans the dev DB ships with | **Not a real numbering scheme.** Cosmetic artefact of a one-off seed; never generated by application code. If you see these in a real tenant the data was hand-injected. The is_test flag work in `members` covers the membership equivalent (see prompt #12 design doc). |
| `L-PROV-…` | Same scaffold (loan side) | Same as above. |

If you need to add a new identifier scheme:

1. Add the sequence as a `<thing>_seq` table with `(tenant_id, year, last_no)` columns (see `services/member/internal/db/migrations/0004_applications.up.sql` — `membership_application_seq` — as the canonical pattern).
2. Pick a 1–4 letter prefix that doesn't collide with any row above. Acronyms beat single letters when the domain isn't obvious.
3. Format `<PREFIX>-YYYY-NNNNN` (5 digits is the default; bump to 6 if you expect >100k records/year/tenant).
4. Update this doc in the same PR.

## Generators

The actual Go generator helpers live alongside their stores:

- `services/member/internal/store/application_store.go::NextAppNoTx`
- `services/member/internal/store/member_store.go::NextMemberNoTx`
- `services/savings/internal/store/loan_application_store.go::nextAppNo`
- `services/savings/internal/store/loan_store.go::nextLoanNo`
- `services/accounting/internal/store/journal_store.go::nextEntryNo`

Each grabs an advisory lock on `(tenant_id, sequence_name)`, increments
the row, and returns the formatted string — so concurrent application
inserts within a single tenant never collide.
