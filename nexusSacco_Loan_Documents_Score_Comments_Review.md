Two existing handlers already INSERT into this table directly:

But there is **no general-purpose upload / list / download endpoint** for officers or applicants to manage application/loan documents.

### Benchmark — how SACCOs and banks handle loan documents

| Pattern | What it means | Where it's used |
| --- | --- | --- |
| **Required-docs checklist** | Per-product: this product requires payslip + bank statement + ID. UI surfaces a checklist; application can't advance to "pending_approval" until every required kind has at least one document attached. | Banks (HF, KCB), modern SACCO platforms (Sasra, Mambu) |
| **Officer captures, applicant uploads** | Hybrid — applicant uploads on member portal; officer can also upload on behalf of applicant from admin UI. Same storage; provenance recorded. | Modern lenders |
| **Versioned documents** | If an applicant re-uploads a payslip (older one expired), keep the prior version. Audit + dispute resolution. | Banks |
| **Document expiry** | Some documents have a usable-by date (KRA PIN cert: 1 year, bank statement: 3 months, valuation report: 24 months). Application UI warns + blocks approval when documents are stale. | Banks |
| **Per-document review state** | Officer marks each doc as "reviewed", "needs replacement", "flagged for fraud". Drives the decision queue. | Bank credit committees |
| **One-click bundle / sharing** | Generate a ZIP of all loan docs for credit committee review or external auditor. | Standard ops feature |

### Recommendation

Build a real Documents tab with:

1. **Upload + list + download endpoints** mirroring the member-documents pattern (`services/member/internal/handler/member.go::UploadDocument` / `DownloadDocument`). Multipart upload to existing storage backend, audit trail, per-tenant size limits.
2. **Required-docs checklist per loan product.** New column `loan_products.required_document_kinds text[]` storing an array of `loan_doc_kind` enum values. UI surfaces this as a checklist on the Documents tab; each required kind shows ✓ (one attached) or ✗ (missing). The workflow approval gate refuses to advance an application that's missing required documents — same pattern as the collateral coverage gate.
3. **Document expiry.** Add `expires_at` column on `loan_documents`. Per-kind default expiry windows in `tenant_operations` (KRA PIN 365d, bank statement 90d, valuation 730d, payslip 60d). UI shows amber chip when expiry within warning window, red after expiry. Approval gate refuses on expired required documents.
4. **Per-document review state.** Add `review_status text DEFAULT 'pending' CHECK (review_status IN ('pending','reviewed','needs_replacement','flagged'))` plus `reviewed_by`, `reviewed_at`, `review_notes`. Officer reviews each doc; status surfaces inline.
5. **Versioning.** When the same `(application_id, kind)` gets a new upload, supersede the prior row with `superseded_by_id` and `is_current=false`. Always show the current row by default; "Show history" reveals prior versions.
6. **Multiple instances per kind.** Some kinds need many documents (e.g. multiple bank statements covering different months). The UI's required-kinds checklist treats a kind as satisfied if **any** current document of that kind exists; the tab also accepts multiple uploads.
7. **Member-portal upload** (when Phase 6's member portal ships) — applicant uploads from their portal; officer sees them appear; verification state stays officer-driven.
8. **One-click ZIP** — `GET /v1/loan-applications/{id}/documents/zip` streams a zipped bundle. Auditor / committee feature.

## 2. Score tab — the "weird string"

### What's there today

Lines 539–555:

```tsx
function ScoreTab({ a }: { a: LoanApplication }) {
  if (!a.scoring_details && !a.scoring_flags) {
    return <div className="empty">No scoring data yet. Hit Re-score in the header to compute.</div>;
  }
  return (
    <div>
      <h4>Scoring details</h4>
      <pre>{JSON.stringify(a.scoring_details ?? {}, null, 2)}</pre>
      <h4>Flags</h4>
      <pre>{JSON.stringify(a.scoring_flags ?? [], null, 2)}</pre>
    </div>
  );
}
```

That `JSON.stringify` is the "weird string" — it's literally raw JSON pretty-printed in a `<pre>` block. Effectively the user is reading developer log output.

**The backing data is rich** — `loan_applications` already has:

- `credit_score int`
- `risk_band text` ('A'..'D' or 'Green'/'Amber'/'Red')
- `affordability_pass boolean`
- `dti_ratio numeric(6,3)`
- `net_disposable_income numeric`
- `computed_max_amount numeric` — multiplier ceiling
- `computed_max_installment numeric` — affordability ceiling
- `recommended_amount numeric`
- `recommended_term_months int`
- `scoring_details jsonb` — per-factor score breakdown
- `scoring_flags jsonb` — hard blocks + advisories

The `Re-score` endpoint already exists: `POST /v1/loan-applications/{id}/score` (handler at `loan_application.go:360`).

### Benchmark — how lenders present scoring

| Pattern | What it shows |
| --- | --- |
| **Headline score + risk pill** | Large number, colour-coded pill (green/amber/red), brief verdict |
| **Affordability waterfall** | Income breakdown → existing obligations → disposable income → proposed installment → DTI ratio. Each step is a row with a number, pass/fail icon. |
| **Multiplier check** | Member's shares × multiplier vs requested amount. Visual gauge or progress bar. |
| **Per-factor breakdown** | Each scoring factor (member tenure, repayment history, employment, etc.) as a row with its contribution to the total score. |
| **Hard blocks vs advisories** | Two distinct sections. Hard blocks need resolution before approval; advisories are warnings only. |
| **CRB summary** (when Phase 6 CRB lands) | Score from each pulled bureau, recent listings, enquiries. |
| **Recommendation card** | "Recommended: APPROVE @ KES 250,000 / 14% / 12 months" or "Recommended: DECLINE — affordability fail". Officer can accept the recommendation with one click. |
| **Score history** | Same applicant rescored multiple times (e.g. after adding guarantors) — timeline of scores. |
| **Re-score action** | One button. Officer clicks → handler runs → updated values populate. |

### Recommendation

Replace the JSON dump with a structured presentation that surfaces every field:

1. **Header card — at-a-glance verdict.**
   ```
   ┌─ SCORING SUMMARY ─────────────────────────────────────┐
   │  Score: 720   Risk: A   [✓ APPROVE recommended]        │
   │  Last scored: 2 hours ago                              │
   │  Recommended: KES 250,000 / 12 months                  │
   │                                       [Re-score]      │
   └────────────────────────────────────────────────────────┘
   ```
2. **Affordability waterfall** as a structured table:
   ```
   Income                                  KES  85,000
     Net salary                            KES  60,000
     Other income                          KES  25,000
   Less: Monthly expenses                  KES (30,000)
   Less: Existing obligations              KES (15,000)
   ─────────────────────────────────────────────────────
   Net disposable income                   KES  40,000   ✓
   Proposed installment                    KES  22,500
   DTI ratio                                       56%   ✓ (≤ 60% policy)
   ─────────────────────────────────────────────────────
   Affordability check                              PASS
   ```
3. **Multiplier check** card:
   ```
   Member shares × 3 (product multiplier)    KES 600,000  ceiling
   Requested amount                          KES 250,000
   Headroom                                  KES 350,000  ✓ within limit
   ```
4. **Hard blocks** (red bordered) and **advisories** (amber bordered) — two distinct sections rendering parsed `scoring_flags`. Each flag has a description, severity, and (if remediable) suggested action. Hard blocks list at top, advisories below.
5. **Per-factor breakdown** — table reading `scoring_details`:
   ```
   Factor                  Weight    Score  Contribution
   Member tenure           20%       80     16
   Repayment history       30%       95     28.5
   Employment stability    15%       70     10.5
   Income consistency      15%       75     11.25
   Existing exposure       10%       80     8
   CRB rating              10%       —      (not yet pulled)
   ──────────────────────────────────────────────────────
   Total                                    74.25 → 720
   ```
6. **Re-score button** is already wired — just present it prominently in the header card with a confirm step.
7. **Score history** — append-only `loan_application_score_history` table (new). Every rescore creates a row. UI shows a sparkline + timeline of past scores so an officer can see "Score went from 580 to 720 after the third guarantor was added."
8. **"View raw JSON"** developer escape hatch — collapsed by default behind a toggle. Sometimes you DO need to see the raw shape; just don't make it the primary view.

For applications that haven't been scored yet, show a clean empty state with the Re-score button and a checklist of what scoring needs (e.g. "Income captured ✓, employment status ✓, guarantors pending — scoring will run with current data; rescore once guarantors confirm").

This recommendation also naturally absorbs the CRB pull display (Phase 6) — when a CRB pull exists, it appears as another factor in the per-factor breakdown.

## 3. Comments tab — design TBD

### What's there today

```tsx
function CommentsTab() {
  return (
    <div className="empty">
      Comments tab — Phase 2 wires the loan_comments table (or audit-
      log-backed thread; design TBD).
    </div>
  );
}
```

So **no table exists yet**, and the design itself is open.

### Benchmark — how lenders use comments

Two distinct purposes that are often conflated:

| Purpose | Audience | Example | Reads like |
| --- | --- | --- | --- |
| **Internal officer notes** | Officers + approvers + auditors only | "Spoke with employer to verify employment — confirmed." | Slack channel for the loan |
| **Member-facing communication** | Officer ↔ applicant | "Please re-upload May payslip — too blurry to read." | SMS / chat thread |
| **Audit annotation** | Auditor reading the file later | "Approver overrode coverage warning — see auth letter attached." | Margin notes |

Most real systems do **both** internal-only and member-facing, with a clear visibility marker on each comment.

### Three design options for Comments

#### Option A — Internal-only notes table

Simplest. Just a list of officer-facing notes. No member visibility. Audit-friendly.

- **Pros**: trivial schema, no permission complexity, fast to ship.
- **Cons**: forces member communication to happen elsewhere (SMS, phone) — the audit trail of officer ↔ member communication lives outside the loan record.

#### Option B — Hybrid: internal + external threads in one tab

Two-tone comment thread: each comment has `visibility` of `internal` or `external`. External comments fire an SMS/email to the member; internal stay in the admin UI. Member portal (Phase 6) shows the external thread; the admin UI shows both with a clear visual marker.

- **Pros**: single audit-traceable channel for all loan-related discussion. Member communication is captured in the loan record, not lost to scattered SMS logs.
- **Cons**: more design surface — visibility rules, member SMS firing, deduplication when the member SMS-replies.

#### Option C — Audit-log-backed thread (no new table)

Reuse the existing audit log. Every "comment" is an audit row of kind `loan.comment_added`. Free.

- **Pros**: no new schema, instant integration with the timeline.
- **Cons**: the audit log is append-only and event-shaped; a free-text comment doesn't fit cleanly. Threading and visibility become contortions. Search becomes harder. Comments mix in with system events ("comment added", "comment added", "loan disbursed", "comment added") — the signal-to-noise ratio degrades.

### Recommendation — Option B (hybrid)

Build a proper `loan_comments` table with visibility marker. The schema:

```sql
CREATE TABLE loan_comments (
  id               uuid PRIMARY KEY,
  tenant_id        uuid NOT NULL,
  application_id   uuid REFERENCES loan_applications(id),
  loan_id          uuid REFERENCES loans(id),                -- one of application_id OR loan_id is set
  parent_id        uuid REFERENCES loan_comments(id),        -- threading
  visibility       text NOT NULL CHECK (visibility IN ('internal','external')),
  body             text NOT NULL,
  attachment_paths jsonb DEFAULT '[]'::jsonb,                -- optional doc attachments
  author_user_id   uuid NOT NULL,
  posted_at        timestamptz NOT NULL DEFAULT now(),
  edited_at        timestamptz,
  edit_history     jsonb DEFAULT '[]'::jsonb,                -- for audit
  pinned           boolean NOT NULL DEFAULT false,           -- pin important comments to top
  member_read_at   timestamptz                               -- for external: when member opened it
);
CREATE INDEX loan_comments_app_idx ON loan_comments (application_id, posted_at DESC) WHERE application_id IS NOT NULL;
CREATE INDEX loan_comments_loan_idx ON loan_comments (loan_id, posted_at DESC) WHERE loan_id IS NOT NULL;
```

The UI:

```
│                                                                    │
│  📌 Pinned (J. Otieno, 2026-05-15)  [Internal]                    │

│  ┌───────────────────────────────────────────────────────────┐   │
│  │ Type a comment...                                          │   │
│  │ ◯ Internal only   ◉ Send to member (SMS + portal)         │   │

│  └───────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────────┘
```

**Behaviours:**

- **Internal comments** appear only in the admin UI. Officers, approvers, auditors see them.
- **External comments**, when posted, fire an SMS to the member with a tokenised link to a single comment thread on the member portal (when Phase 6 ships) or a simple read-only public web page (interim). The member can reply via SMS-back (if the inbound SMS hook is available) or via the portal.
- **Audit log** records every comment as an event for the timeline. The Comments tab is the readable view; the audit log is the immutable record.
- **Permissions**: `loans:comment:internal` (post internal), `loans:comment:external` (post external + send to member). Defaults: all officers can post internal; only credit officers + above can post external.
- **Pinning**: any officer can pin the comment they consider critical (e.g. "do not disburse until X"). Pinned comments stay at the top.
- **Editing**: edits preserve history. The UI shows "(edited)" with a hover to see prior versions.
- **Searchable**: free-text search across comment bodies, scoped per application/loan. The same search index can later cover the Documents tab's notes.
- **Member-facing**: when Phase 6 lands, the member's loan portal shows the external thread as a Slack-like channel. Until then, an SMS-only flow keeps members informed; they reply via SMS.
- **Templates**: a small set of pre-written comment templates ("Documents needed", "Approval delay", "Disbursement scheduled") — quick clicks for common officer messages.

This is the only one of the three tabs that has a **real design decision** behind it; Documents and Score are mostly "build the UI." The hybrid model is a small amount more work than internal-only but it captures the actual workflow.
